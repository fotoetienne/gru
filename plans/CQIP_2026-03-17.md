## Section 2: Testability Improvements

### 2.1 Dependency Injection Opportunities

#### 2.1a No GitHubClient trait — all GitHub calls hardcoded to `gh` CLI
**Effort: L**

Every GitHub operation in `github.rs` (930 lines) is a free function calling `gh_cli_command()` directly with no abstraction layer. The same pattern appears throughout:
- `ci.rs`: `fetch_check_runs` (lines 259-352), `fetch_check_logs` (lines 356-445) both call `github::gh_cli_command` directly
- `merge_judge.rs`: `get_pr_fingerprint` (lines 169-246) and `fetch_pr_context` (lines 258-353) each make ~4 parallel `gh_cli_command` calls
- `minion_resolver.rs`: `scan_all_minions` (lines 235-342) uses `std::process::Command::new("git")` inline

**Recommendation:** Extract a `GitHubClient` trait analogous to the existing `AgentBackend` pattern (`src/agent.rs`). Core methods: `get_issue`, `post_comment`, `add_label`, `remove_label`, `list_check_runs`, `get_pr`, `list_pr_reviews`, `create_pr`. Inject via function parameter or struct field. A `GhCliClient` concrete type wraps the existing `gh_cli_command` calls.

#### 2.1b No ProcessSpawner trait — subprocess calls hardcoded
**Effort: M**

Four locations spawn subprocesses with no injection point:

| Location | Binary | Function |
|---|---|---|
| `ci.rs` ~line 571 | `"claude"` | `invoke_ci_fix` |
| `commands/fix/mod.rs` ~line 55 | `gru_exe` (via `std::env::current_exe()`) | `spawn_worker` |
| `commands/fix/monitor.rs` ~line 105 | `"gru"` | `trigger_pr_review` |
| `merge_judge.rs` ~line 482 | `"claude"` | `invoke_judge_cli` |

**Recommendation:** A minimal `trait ProcessSpawner { fn spawn(&self, cmd: Command) -> Result<Child> }` with a `RealSpawner` passthrough and a `MockSpawner` for tests.

#### 2.1c Global singletons block parallel testing
**Effort: M**

- **`workspace.rs:GLOBAL_WORKSPACE: OnceCell<Workspace>`** — Once initialized, all tests in the same process share a single workspace root. Parallel test runs that touch different worktrees will collide. Forces `#[serial]` test annotations or temp-dir hacks.
- **`minion_registry.rs:MinionRegistry::load(None)`** — The `None` path always resolves to `Workspace::global()`. Tests that need isolated registries must fight the singleton.
- **`config.rs:try_load_config()`** — Reads from `~/.gru/config.toml` on disk at every call. No injection mechanism; tests cannot supply a synthetic config without writing to the real filesystem.
- ~~**Two `static WORKSPACE: Lazy<...>` instances** in `src/minion.rs:11` and `src/minion_registry.rs:14`~~ [RESOLVED 2026-03-19 — both now call `Workspace::global()` directly]

**Recommendation:** The `GLOBAL_WORKSPACE: OnceCell` singleton itself remains. Add `#[cfg(test)]` override via `thread_local!` or `once_cell::sync::OnceCell::set`. Add a `Config::from_path(path)` constructor that bypasses the global default.

#### 2.1d Business logic entangled with I/O
**Effort: M**

- **`commands/fix/agent.rs:run_agent_session_inner`** (lines 125-276, ~151 lines): Phase detection heuristics that string-match on "Plan", "Implement", "test" are embedded inline in a closure. PID registration, token usage cleanup, and GitHub comment posting are also inline.
- **`agent_runner.rs:log_event`** (lines 134-150): Direct `tokio::fs::OpenOptions` call with no filesystem abstraction.
- **`ci.rs:monitor_and_fix_ci`** (lines 730-873, ~143 lines): CI polling, failure analysis, Claude invocation, `git push`, and retry loop interleaved with no seams.
